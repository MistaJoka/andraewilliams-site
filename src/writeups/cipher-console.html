Cipher Console

The first tool on this site that runs my own backend code: a Python serverless function on Vercel. This log documents how it was built with AI codegen — the brief, what the model produced, what I changed, and what I took away.

The brief

I wanted my first piece of server-side code on the site — proof I can ship Python, not just static HTML. The brief I gave the AI: a tactical "Cipher Console" that encodes and decodes text with four reversible transforms (Base64, Hex, ROT13, URL), where the work runs in Python on a Vercel serverless function rather than the browser, because running it server-side is the whole point. Keep it standard-library only — no dependencies, free to run — and match the site's existing command-center styling.

What the AI generated

  • A Python function at api/cipher.py using Vercel's BaseHTTPRequestHandler pattern, with a pure transform() core and a thin HTTP wrapper around it.
  • A front-end page (cipher-console.html + cipher-console.js) that POSTs { op, mode, text } and renders the result or a structured error.
  • Wiring into the site: a manifest entry, a Tools card, and this build log.

What I changed / verified

  • Separated logic from transport. The transforms live in a pure transform() function so they can be unit-tested with plain python3 — known-answer checks (hex of "abc" is 616263, ROT13 of "abc" is "nop", a bad-Base64 decode raises instead of crashing) all pass before the function ever deploys.
  • Errors return 400, never 500. Invalid input (e.g. decoding malformed Base64) is caught and returned as { "error": … }, which the UI shows as a red ERROR // line instead of a stack trace.
  • ROT13 hides the mode toggle. It's self-inverse, so encode/decode is meaningless — the UI drops the control instead of showing a no-op.
  • Confirmed the Vercel integration. The one real risk was whether an /api/*.py function deploys alongside the site's custom build (outputDirectory: _site). It does.
What I learned

The leverage in AI codegen isn't "type a prompt, ship the output" — it's directing the shape. Insisting the transforms be a pure, testable function (not tangled into the HTTP handler) is what made it verifiable with three lines of python3. And the jump from static HTML to a serverless function turned out to be small: one file in /api, standard library only, no new infrastructure. That's the on-ramp to the server-based tools I'm building toward.
