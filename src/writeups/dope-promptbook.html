DOPE Prompt Book

A standalone prompt library and composer — not a page on this site. Snips, recipes, and kits assemble into a working prompt you can copy or reuse. The portfolio links here; the app owns its UX.

Problem

Reusable AI prompts live everywhere — notes apps, gists, chat history, random docs. None of that scales when you want to find, filter, and compose prompts the same way every session. Copy-paste from scattered files is slow; editing in-place loses version context.

DOPE is the answer on my side of the keyboard: a dedicated surface for prompt pieces (snips), structured bundles (recipes), and reusable assemblies (kits), with a composer that stacks them into one output.

Approach

Split the portfolio from the product. andraewilliams.com stays static and operator-themed — proof, status, case files. The prompt book runs as its own Next.js app on a subdomain so it can grow (auth, storage, search) without dragging the main site into a framework.

  • Library — browse and filter prompt pieces by type and recency.
  • Composer — stack selected pieces; empty state tells you to open the library.
  • Snips / Recipes / Kits — three layers of reuse from atomic text to full workflows.
  • Search — desktop shortcut / for fast lookup without leaving flow.
  • Public demo — read-only mode so visitors can explore without signing in.

Architecture (high level)

Client-heavy UI with Next.js App Router. Prompt entities are modeled as typed pieces the library can sort (recently edited, recently used, most used, alphabetical) and filter. The composer holds an ordered list of references to those pieces and renders the combined prompt for copy/export.

Local-first is the target posture: your library should survive offline and sync when connected. The public demo is read-only today; private write paths are the next maturity step.

Tradeoffs

  • Subdomain vs embedded iframe — separate deploy and DNS, but clean boundaries and independent release cadence.
  • Read-only demo — safe for public traffic; hides the full create/edit loop until auth and persistence are ready.
  • Next.js on Vercel — not free-tier-zero forever at scale, but fits the “real app” bar this project sets for itself.

What I'd change next

  • Authenticated write path — private library sync, not just demo browsing.
  • Export formats — markdown bundle, JSON kit, clipboard with metadata.
  • Case file screenshots — capture library + composer states for this writeup.
  • Link back to portfolio mission status when a prompt ships a site feature.
